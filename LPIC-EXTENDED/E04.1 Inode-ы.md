```
Индексная нода (inode) - структура данных файловой системы, содержащая метаданные файла, но не его имя.
```

Каждый файл в Linux имеет inode, а файловая система представляет собой соответствие:

filename -> inode -> data blocks

# Содержимое inode

Inode хранит:

- права доступа (rwx)
- количество ссылок (link count)
- UID/GID владельца
- размер файла
- timestamps (atime, mtime, ctime)
- указатели на блоки данных

Команда `ls -li` показывает расширенный `ls` с inode-номером файла.

Inode не хранит в себе имя и путь к файлу, т.к. они находятся в записях каталога (directory entry), которые представляют собой таблицу имён файлов относительно нод.

# Связь нод и ссылок

Поскольку inode является идентификатором файла, ссылки работают через него, а именно:

## Hard links

При создании жестких ссылок, оба имени указывают на одну ноду.

Особенности жестких ссылок:
- одна нода
- увеличивается кол-во ссылок 
- общие данные
- удаление одного имени не удаляет файл  
- работает только внутри одного раздела

Удаление файлов, при жесткой связке, уменьшает кол-во ссылок соответственно, а сам файл исчезает, если кол-во ссылок равно нулю.

## Soft links

Мягкие ссылки являются отдельным файлом, содержащий пусть к другому файлу

Особенности мягких ссылок:
- отдельная нода от файла-источника
- хранит путь к файлу-источнику
- может ссылаться на другой раздел
- легко сломать (например, при перемещении файла)

# Ноды и удалённые файлы

Файл может существовать без имени, т.к. у каждого файла есть свой файловый дескриптор, и пока процесс, использующий этот файл, держит его в своей памяти, нода остаётся и продолжает "давать доступ к файлу"

Если df показывает, что файл занимает место, когда самого файла нет (du), то произошел вышеописанный случай, и дабы его продиагностировать, можно воспользоваться командой `lsof | grep deleted`. 

LSOF - list open files, 
deleted - один из статусов файла, по которому осуществляется фильтрация.
